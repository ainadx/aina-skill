# AiNa App Factory — build

Take a single request and produce a complete product package. The **headline
deliverable is the document set** — one detailed markdown doc per stage — with a
working codebase alongside. **Your local Claude is the engine**: you execute each
persona yourself; no server and no API key are required.

## Setup

- `PLUGIN_ROOT` = `${CLAUDE_PLUGIN_ROOT}` — this plugin's directory.
- `PY` = `python3` (the helper scripts are stdlib-only; no venv).
- **Personas** (the IP that writes the docs): `$PLUGIN_ROOT/personas/<stage>/*.md`,
  run in their `order:` frontmatter.
- **Scripts**: `$PLUGIN_ROOT/scripts/{classify_stack,seed_starter,package_output}.py`.
- **Config** (from plugin userConfig, with `aina-build.yaml` in the CWD overriding):
  - `${CLAUDE_PLUGIN_OPTION_ENGINE}` — `local` (default) | `openrouter`
  - `${CLAUDE_PLUGIN_OPTION_OPENROUTER_API_KEY}` — only for engine=openrouter
  - `${CLAUDE_PLUGIN_OPTION_OUTPUT_MODE}` — `folder` | `zip` (default) | `git-repo`
  - `${CLAUDE_PLUGIN_OPTION_GITHUB_ORG}` — owner for git-repo mode (blank = personal)
  - `${CLAUDE_PLUGIN_OPTION_DEPLOY_TARGET}` — `none` (default) | `vps` | `vercel`
  - `${CLAUDE_PLUGIN_OPTION_VPS_HOST}` — `user@host` for deploy_target=vps

## Engine

- **local (default):** YOU are the engine. Read each persona file and produce its
  output yourself, threading prior stages' outputs forward. No API key, no cost
  beyond the user's own Claude session.
- **openrouter (optional):** if engine=openrouter and a key is set, you MAY offload
  the bulk analyst personas to a cheaper OpenRouter model (POST to
  `https://openrouter.ai/api/v1/chat/completions` with the key) to save the user's
  Claude quota; keep synthesizers + the dev build on your own (Claude) reasoning.
  If no key is present, silently fall back to local.

## Output location

Create `./aina-out/<slug>/` with:
```
docs/    ← the document set (the winner) — one file per stage
app/     ← the generated codebase (seeded golden starter, filled in)
README.md
```
Create `docs/` first and write each stage's document into it as you go, so the
package is valuable even if the user stops early.

## 1. Intake (braindoc) — HITL

If the request lacks detail, ask the user a short, sharp round (use
AskUserQuestion): what it does, who it's for, must-have features, constraints.
Then write **`docs/01-braindoc.md`** — the framed problem, users, scope, risks.

## 2. Decide the stack — auto or operator choice

Run the classifier on the brief:
```
$PY "$PLUGIN_ROOT/scripts/classify_stack.py" --text "<brief + features>" --archetype <archetype>
```
It returns `lean` + `profile` (backend → FastAPI+HTMX → vps; frontend → Next.js+
shadcn → vercel). Confirm with the user if they're present, else take the default.

## 3. Generative stages — the loop (each writes a document)

Run these stages in order; each one is a gate (summarize, let the user
approve/edit before continuing). For each stage, read
`$PLUGIN_ROOT/personas/<stage>/*.md` (sorted by `order:`), execute each persona,
thread outputs forward, and **write the stage document**:

| Stage | Personas dir | Document |
|-------|--------------|----------|
| intent     | `personas/intent`     | `docs/02-intent-sow.md` |
| ideation   | `personas/ideation`   | `docs/03-ideation-prd.md` |
| dev        | `personas/dev`        | `docs/04-dev-notes.md` (+ the code, see below) |
| security   | `personas/security`   | `docs/05-security.md` *(optional)* |
| readiness  | `personas/readiness`  | `docs/06-readiness.md` *(optional)* |
| marketing  | `personas/marketing`  | `docs/07-marketing.md` (+ `app/marketing.html`) |
| docs       | `personas/docs`       | `docs/08-user-docs.md` *(optional)* |
| ops        | `personas/ops`        | `docs/09-ops-runbook.md` |

(Default set: intent, ideation, dev, marketing, ops. Include security/readiness/
docs if the user asks or the project warrants it.)

### Dev stage — seed then extend, then verify

1. Seed the golden starter for the chosen framework into `app/`:
   ```
   $PY "$PLUGIN_ROOT/scripts/seed_starter.py" ./aina-out/<slug>/app fastapi
   ```
   (FastAPI ships a professional shell — sidebar/design-system/HTMX — so it can't
   render bland.)
2. Implement the real app by **extending** the starter (follow
   `$PLUGIN_ROOT/personas/dev/01-scaffolder.md` + `03-frontend-builder.md`): reuse
   the design-system classes, add real routes/models/pages, keep `/roadmap`. Never
   re-emit the base CSS/layout, never ship a stub page.
3. **Verify** (the terminal advantage): actually run it
   (`python3 -m uvicorn app.main:app --port <free>` from `app/`, curl `/healthz`
   + pages) and FIX failures before moving on.
4. Write `docs/04-dev-notes.md` (architecture, how to run, what's stubbed).

## 4. Package the output

```
$PY "$PLUGIN_ROOT/scripts/package_output.py" ./aina-out/<slug> ${output_mode} \
    --org "${github_org}" --private
```
- `folder` → leaves it on disk · `zip` → also writes `<slug>.zip` · `git-repo` →
  `gh repo create` + push (needs `gh auth login` done by the user).

## 5. Deploy (optional — only if deploy_target != none)

- `vps`: build the app's Docker image and run it on `${vps_host}` over SSH on a
  free port; report the URL. (Requires the user's own server + SSH access.)
- `vercel`: `vercel --prod` for a Next.js app.
Default is `none` — code + docs are the product; deploying is a bonus.

## 6. Final summary

Print plainly:
- **Docs:** list `docs/*.md` (the document set)
- **Code:** `aina-out/<slug>/app/`
- **Package:** the zip path or repo URL
- Stack chosen + anything skipped.

## Principles

- **Documents are the product.** Every stage must leave a real, detailed markdown
  doc in `docs/` — that's the value, the code is one of the outputs.
- **The installer's Claude is the engine.** No author server, no required API key.
  OpenRouter is an optional accelerator, not a dependency.
- **Self-contained.** Everything needed (personas, starters, scripts) ships in this
  plugin; reference it via `$PLUGIN_ROOT`. No external repo, no hardcoded hosts.
- **Honor the golden-stack guarantees:** no bland output, no stub pages, one design
  system, the right stack for the job.
- **Verify before you ship.** Run and fix the app; don't one-shot it.
